MySQL中的锁

发表于2021-07-11|更新于2021-07-11|DevelopMySQL
|字数总计:6.9k|阅读时长:22分钟|阅读量:

​ MySQL 中的锁一直都是被人津津乐道的一点。锁对于MySQL来说是非常重要的一环，是用来控制并发安全中非常重要的一环。在MySQL中用来解决并发安全问题的有三大步骤。事务隔离机制、锁机制、MVCC多版本并发控制隔离机制

1、锁

1.0、MVCC中的当前读和快照读

​ 在MySQL单一的事务隔离级别中，很容易就发现存在非常大的问题，脏读幻读等现象会得不到保障，下面我们就来看一下，MySQL在解决这一系列问题中，锁起到了哪些至关重要的作用。

​ 下面主要以InnoDB引擎来分析MySQL锁机制。因为MYISAM引擎不支持事务。所以在做操作时，直接锁的整张表。

​ 在InnoDB中实现是基于多版本的并发控制协议（MVCC）Multiversion concurrency control与之相对应的就是一种加锁机制。MVCC最大的好处就是在于将读写分离开，一些本身就是原子操作的操作根本就不会加锁。极大的提高了并发性能。这里就引出了一个概念就是，MySQL的读不加锁，读写不冲突的高级特性。在如今读多写少的OLTP应用中，读写不冲突是非常重要的。这就是为什么现在市面上绝大多数的RDBMS，都支持MVCC。

​ 在MVCC并发控制中，将读分为了两大类，快照读（snapshot read）与当前读（current read）。

  • 快照读，读取的是记录的可见版本（有可能是历史版本，这设计到了MySQL的MVCC基于事务id的实现细节。通俗的讲就是快照读读取的数据是所有未参与正在执行其他事务线程的可见内容。），不用加锁。
  • 当前读，读取的是记录的最新版本（这里的版本就是MVCC控制中的提交事务的的最新id），并且，当前读返回的记录，都会加上锁，保证其他事物不会再并发修改这条记录。

1.1、在InnoDB中，哪些操作属于快照读，哪些操作有属于当前读呢?

  • 快照读

    • 简单的select操作，属于快照读（查询操作本身就是原子操作），不加锁（当然存在例外）

      • select * from table where ？ ;

  • 当前读

    • 特殊的操作，插入/更新/删除操作，属于当前读（会影响数据的，非原子操作），需要加锁

      • select * from table where ? lock in share mode;（添加共享锁）
      • select * from table where ? for update;
      • insert into table values (…);
      • update table set ? where ?;
      • delete from table where ?;

      所有以上语句都是属于当前读，需要读取MySQL中的最新版本。并且，读取之后，还需要保证其他并发事务不会修改当前纪录，对读取纪录加锁，其中，除了第一条语句，读读取纪录加S锁（共享锁）外，其他操作，都加的是X锁（排它锁）

共享锁和独占锁的区别

SELECT … LOCK IN SHARE MODE走的是S锁(意向共享锁)，即在符合条件的rows上都加了共享锁，这样的话，其他session可以读取这些记录，也可以继续添加IS锁，但是无法修改这些记录直到你这个加锁的session执行完成(否则直接锁等待超时)。

SELECT … FOR UPDATE 走的是X锁(意向排它锁)，即在符合条件的rows上都加了排它锁，其他session也就无法在这些记录上添加任何的S锁或X锁。如果不存在一致性非锁定读的话，那么其他session是无法读取和修改这些记录的，但是innodb有非锁定读(快照读并不需要加锁)，for update之后并不会阻塞其他session的快照读取操作，除了select …lock in share mode和select … for update这种显示加锁的查询操作

所谓的共享锁，就是在对命中的数据加共享锁时，其他的session还可以对其进行加共享锁。并且也可以读取该行数据

在多个session 同时对同样的数据加共享锁时，所有的session 都可以加锁成功，并且在对数据进行读取操作时，都不会影响也是都可以成功的，但是如果多个session 对同样的数据做其他当前读操作。那么就会引发锁等待，并且会出现死锁现象。

所以共享锁存在非常大的弊端，在InnoDB的MVCC中已经实现了快照读的不加锁读。那么很大程度上不需要共享锁了。在需要加锁的地方建议使用for update 加上x锁

1.2、为什么插入/删除/更新操作都归位了当前读？

​ 原因很简单，就是在做上述操作时，MySQL必须先定位到数据，所以这一步就涉及到了读取数据。然后再对数据进行非原子操作，即为当前读操作。为了保证该操作的并发安全性，必须加锁。

​ 简单看一下一条update在MySQL中的执行流程

从图中，可以看到， 一个 Update 操作的具体流程。 当 Update SQL 被发给 MySQL 后， MySQL
Server 会根据 where 条件，读取第一条满足条件的记录，然后 InnoDB 引擎会将第一条记录
返回，并加锁 (current read)。 待 MySQL Server 收到这条加锁的记录之后， 会再发起一个
Update 请求，更新这条记录。 一条记录操作完成，再读取下一条记录， 直至没有满足条件
的记录为止。 因此， Update 操作内部，就包含了一个当前读。 同理， Delete 操作也一样。 Insert
操作会稍微有些不同， 简单来说，就是 Insert 操作可能会触发 Unique Key 的冲突检查，也会
进行一个当前读。

根据上图的交互， 针对一条当前读的 SQL 语句， InnoDB 与 MySQL Server 的交互，是一
条一条进行的，因此， 加锁也是一条一条进行的。先对一条满足条件的记录加锁，返回给
MySQL Server，做一些 DML 操作；然后在读取下一条加锁，直至读取完毕。

1.3、InnoDB的加锁方式

​ 在MySQL中，InnoDB引擎的加锁方式是加载索引上的，所以在执行一条当前读SQL时，如果当前操作的字段并非索引字段，那么很有可能行锁会直接升级为表锁。

​ 因为InnoDB的索引特性设计为聚簇索引，是一个完成的记录。通过主键索引，就可以获取记录所有的列。

1.5、临键锁（Next-key Locks）

​ Next-Key Locks是行锁与间隙锁的组合。像上面那个例子里的这个(3,11]的整个区间可以叫做临键锁。

临键锁就是gap锁的一个整体，而gap锁则是一个细粒度的叫法。

1.6、2PL：Two-Phase-Locking

​ 两阶段锁。这是传统的RDBMS加锁的一个原则，其含义就是将锁操作分为两个阶段：加锁阶段与解锁阶段，并且保证加锁阶段与解锁阶段不相交。

​ 下面对加锁与解锁阶段做一个简单的图示

如图所示，2PL是将加锁和解锁两个不走完全的分离开了，加锁阶段只加锁，不解锁，而解锁阶段只解锁，不加锁。

1.7、综上总结优化

  1. 针对InnoDB的加锁对象（只对索引数据加锁。），所以尽量操作当前读时，操作含索引的字段，避免行锁升级为表锁
  2. 合理设计索引，尽量缩小锁的范围
  3. 尽可能减少检索条件范围，避免gap锁范围太大。（媲美表锁的大小之后，与表锁无差别）
  4. 尽量控制事务大小，减少锁定资源和时间长度，涉及事务加锁的sql尽量放在事务最后执行

以上两条SQL的加锁状态进行分析应该得到什么答案呢？

SQL1：不加锁，因为执行的是快照读

SQL2：加锁，因为是当前读，对id = 11 的记录加上写锁（走主键索引）

看上去上述的两条结论没有什么大的问题，并且貌似也是对的。但是如果一个人突然扔给你这两个SQL，询问你加锁状态，那么这个提问题的人一定非常的业余。原因就是上述两条SQL存在非常多的不确定性，因为有非常多的前提需要注意，比喻说该SQL是在什么存储引擎下？又是什么隔离级别？id是否为主键呢？有是否是索引呢？ 这些前提极大程度的影响了MySQL对其加锁的具体操作

接下来就一下集中前提做简单分析

  1. id列是不是主键？
  2. 当前系统的隔离级别是什么？
  3. id列如果不是主键，那么id列上有索引吗？
  4. id列上如果有二级索引，那么这个所以是唯一索引吗？
  5. 两个SQL的执行计划是什么？索引扫描？全表扫描？

以下组合都建立在这样的一个假设中，如果id 是索引，那么sql 一定会走索引扫描。（其实具体的情况非常的复杂，具体还是得看MySQL的真正执行计划）

下面就上述前提，给出几种组合，来逐一分析具体的加锁情况

 组合一： id 列是主键， RC 隔离级别

 组合二： id 列是二级唯一索引， RC 隔离级别

 组合三： id 列是二级非唯一索引， RC 隔离级别

 组合四： id 列上没有索引， RC 隔离级别

 组合五： id 列是主键， RR 隔离级别

 组合六： id 列是二级唯一索引， RR 隔离级别

 组合七： id 列是二级非唯一索引， RR 隔离级别

 组合八： id 列上没有索引， RR 隔离级别

 组合九： Serializable 隔离级别

其实在上面的八种组合中，第一条sql就是单纯的快照读，不会加任何锁。所以在前面的八个组合中重点分析sql2

2.1、id为主键+RC隔离级别

​ 这种组合是最简单的，在Read Committed 隔离级别 下，ID为主键，给定SQL: DELETE FROM account WHERE id = 11;只需在主键上 id = 11 这行记录上加上X锁即可。如下图：

结论：id为主键时，此SQL值需要在id=10这条记录上加上X锁即可。

2.2、id 列是二级唯一索引+RC 隔离级别

​ 跟上一个组合不同的是，id并非主键索引，而是一个Unique的二级索引。那么在RC下：DELETE FROM account WHERE id = 11; 会加什么样的锁呢？

当前组合中，查询条件为非聚簇索引，是一个二级唯一索引，索引MySQL在定位数据的不走分为如下两步

  1. 先根据唯一索引id定位到id = 11的索引记录，并且将该索引记录加上X锁。
  2. 然后通过二级索引的叶子节点指向的主键索引，定位到name = ‘g’ 的索引记录，然后将该索引记录加上X锁。

为什么在id不是主键索引的情况下，要加上把锁呢？

​ 假设，当前sql在执行的同时，有另外一个session正在通过主键索引修改id = 11 的值。那么如果不加锁。两个当前读操作相互感知不到，违背了同一条记录上的更新/删除需要串行化执行的约束。

结论：若id不是主键索引的情况下，而是一个唯一索引，那么就需要将将加两把X锁，分别将唯一索引的记录和主键索引的记录都锁上。

​ 并且不用考虑其他索引的情况，因为不管什么索引，最后操作的都是主键索引对应的记录。

2.3、id非唯一索引+ RC 隔离级别

​ 与上面两种情况不同的是，此时的id为非唯一索引。所以在此时的id 可能会存在多个。DELETE FROM account WHERE id = 11 的加锁方式如下图

从上图我们可以很简单的看出来，当id为非唯一稀疏索引时，MySQL在执行这样一条当前读SQL时，会先给所有匹配的非唯一稀疏索引加上X锁，然后在由这些加锁的索引找到对应的主键索引，然后将其加上X锁。

总结：若id列上有非唯一索引，那么对应的所有满足SQL查询条件的记录，都会被加锁。同时，这些记录在主键索引上的记录，也会被加锁。

2.4、id 列上没有索引+ RC 隔离级别

​ 这个组合就更加过分了，id列上的最后一种可能性，就是没有索引。那么前面就有铺垫到，MySQL InnoDB的加锁都是加载索引上，如果操作的列上没有索引。那么该加锁会从行锁升级为“表锁”。

​ 那么在RC下，DELETE FROM account WHERE id = 11 ，id = 11 这个where条件在没有索引的情况下只能走全表扫描，这种加锁情况如下图：

从上图可以看到，当操作一个列上不存在索引的SQL时，会引发类似于表锁的情况。可能有的人就会好奇为什么MySQL不只将命中的记录加锁呢？

​ 这是跟MySQL的底层实现有关的。InnoDB 如果在一个条件不能快速通过索引定位到的话，存储引擎层面会将所有记录加锁后返回，然后又MySQL Server层进行过滤。因此也就把所有的记录都锁上了。

​ 其实可以理解为，MySQL的底层加锁操作全是有存储引擎来完成的。

结论：若 id 列上没有索引， SQL 会走聚簇索引的全扫描进行过滤，由于过滤是由 MySQL Server层面进行的。因此每条记录，无论是否满足条件，都会被加上 X 锁。但是，为了效率考量，MySQL 做了优化，对于不满足条件的记录，会在判断后放锁，最终持有的，是满足条件记录上的锁，但是不满足条件的记录上的加锁/放锁动作不会省略。同时，优化也违背了 2PL的约束。

2.7、id 列是二级非唯一索引+ RR 隔离级别

​ 当前组合是RR级别下最特殊的一种情况。上述两种情况由于约束的比较死，id列仅且只有一个能存在。所以加上一把X锁就可以规避掉所有的情况。

​ 周所周知，MySQL支持四种隔离级别，在InnoDB引擎中，在RC及以上的隔离级别是允许出现幻读的。但是在InnoDB中RR及以下是不允许出现幻读的。在组合七种就是他发挥作用的地方了。

​ 由于id列是一个二级非唯一索引。那么 DELETE FROM account WHERE id = 11的weher条件既有可能命中多条（我们暂且认定为id 是一个顺序的列）

​ 这里就是引入了GAP锁。GAP锁顾名思义就是一种间隙锁。为了防止在当前session操作时，其他session添加满足当前session条件的列，造成幻读情况。

​ 举个例子，假设在session 1操作delete操作时，与此同时，session2 添加添加了一条数据【11，ff，xxx】，那么词条数据肯定是符合B+树的数据特性的前提下，也是满足session1 的条件的，此时的session2的添加操作将会被GAP锁阻塞住。这就是GAP锁的作用

​ 在上文提到的临键锁，就是GAP锁加上行锁的一个整体称呼！

需要特别注意的是：双R下的解决幻读有非常大的争议。后面详细再说

总结：在双R的隔离级别下，加锁的步骤会比较复杂，MySQL会考虑到所有复杂的情况，在定位符合where id = 11 时，定位到第一条服务条件的记录时，会给该记录加上X锁，同时也会在该记录的左右添加GAP锁。然后再在该记录所对应的主键的聚簇索引上加上X锁，然后再匹配下一条记录。知道找到所有符合条件的记录。

​ 这里会有一个疑问，既然MySQL的锁是加上索引上的，那么此时的GAP锁只是加在了id索引上的，那么如果通过其他索引插入满足where id = 11 的记录，应该是会成功的。那么在 DELETE FROM account WHERE id = 11的session 中是否会读到这条记录呢？

答案是读取不到的。这也是解决幻读的另外一个非常重要的手段。MVCC机制。后面详细说明

2.8、id 列上没有索引+RR 隔离级别

​ 在组合八种，id没有索引，考虑到并发安全性问题，Innodb会将上锁的所有行都加上GAP锁。上锁的情况会跟[2.4、 id 列上没有索引+ RC 隔离级别](#2.4、 id 列上没有索引+ RC 隔离级别)一样。但是更加恐怖的是在所有锁上面会都加上GAP锁。如图：

从上面的图示就可以看出来是一个非常恐怖的事情，操作没有索引的id列时，会进行全表扫描。在扫描的过程，会将所有扫到的列加上x锁，与此同时也会加上GAP锁。设想一下，如果是一张千万级别的大表，做如此加锁操作。性能消耗得多么恐怖。

​ 当然与组合四一样，MySQL对这种情况实际上也作出了对应的优化，就是semi-consistent read，在semi-consistent read开启的情况下，对于不满足查询条件的记录，MySQL会提前释放锁。针对DELETE FROM account WHERE id = 11这种情况。MySQL最后的加锁情况会跟组合七一样。但是semi-consistent read开启也会对MySQL性能造成一定程度上的影响

​ 关于semi-consistent read，后续详解

总结：在双R级别下，操作一个没有索引的列，会引发非常恐怖的情况发生，首先会造成全部的列加上X锁（相当于表锁了），并且带上这些列之间的GAP锁。用来杜绝所有的并发当前读（更新/插入/删除操作）。当然MySQL也为此带来了优化。就是semi-consistent read，但是semi-consistent read也会带来其他的问题，因此MySQL官方不建议使用。

2.9、Serializable 隔离级别

​ 在当前隔离级别下DELETE FROM account WHERE id = 11与双R下的加锁情况是一致的。但是重点是，在该隔离级别下，很多规范发生了变化，比喻说MySQL的MVCC 多版本的并发控制协议变成了基于锁的并发控制协议（Lock-Based Concurrency Control）。在 Serializable 隔离级别就再也不存在当前读和快照读的区分了，所有的操作都将编程串行化的了。

​ 所以在前面提到的SQL1SELECT * FROM account WHERE id = 11;也会加锁。

总结：在MySQL的InnoDB引擎中，所谓的读不加锁并不是适用于所有的场景，而是与隔离级别密切相关的。在Serializable 隔离级别下，这种情况将不复存在。所有的操作都会串行化，加锁执行。也可以理解为，所有的操作都是当前读

3、一条复杂的SQL分析

​ 下面来分析一下在日常工作中遇到的常规SQL。带聚合索引的SQL加锁情况。

​ 从上图的表结构可以看出idx_t1_put是一个联合索引。联合索引需要满足最左前缀原则。在分析加锁情况之前，首先要弄清楚where条件语句中哪些是走索引，哪些又不能够走索引。

  1. Index key ：puttime > 1 and puttime < 20满足最左前缀匹配规则。则puttime字段是走idx_t1_put索引的。
  2. Index Filter：userid = 'hdc'，由于最左匹配原则是，在前面的判断规则中如果存在范围判断，后续的匹配将不会走索引。所以userid字段时可以在idx_t1_put上进行过滤，但是不会走索引，不属于index key
  3. Table Filter：comment is not null。此条件不属于任何索引。只能在主键的聚合索引下过滤。

当前分析在MySQL的默认隔离级别下进行（双R）

上图的加锁情况分为两种，由于MySQL的版本不同，可能出现不同的情况。

由于在MySQL5.6之前，不支持Index Condition Pushdown (ICP),因此Index Filter 是在MySQL的Server层进行过滤的。所以在InnoDB存储引擎的层面，该行记录是会加上X锁的（图中用红色的箭头标记出来了）

​ 但是在5.6之后，MySQL支持ICP，Index Filter 是在存储引擎层面进行过滤的，所以在对Index Filter 处理之后，上图中红色箭头不会加X锁。

​ 然后再分析Table Filter 时，是在主键的聚合索引中读取后，在MySQL Server层进行过滤的。跟上述的[2.4、 id 列上没有索引+ RC 隔离级别](#2.4、 id 列上没有索引+ RC 隔离级别)情况一样。所以在存储引擎层，还是会加上X锁，在MySQL Server 层才会对不满足要求的列进行释放锁。最后获得满足要求的列。

​ 虽然最后我们获得一条满足SQL的记录，但是可以看到加锁的数量，是远远超过命中的记录数量的。

总结：在双R 隔离级别下。针对一个复杂的SQL，首先需要提取其where条件。确定好where 条件中具体走索引的情况，进行具体分析加锁情况。

​ 总之，MySQL中的加锁都是基于索引进行的。所以在分析加锁时，首先要明确好索引的情况。还有就是MySQL的加锁是以隔离级别来划分能力的。所以在具体的加锁实现细节上，前提也要确定好当前sql 是跑在什么样的隔离界别下的。
